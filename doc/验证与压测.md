# 验证与压测

这份说明对应最后一个阶段，用来验证目前已经补上的 7 个阶段没有明显回退。

## 前置条件

1. 启动基础依赖：

```powershell
docker-compose up -d mysql redis rabbitmq kafka
```

2. 启动服务：

```powershell
cd cmd
go run .\main.go
```

3. 准备一个可登录用户，以及这个用户的：

- 用户名
- 登录密码
- 6 位支付 key
- 一个可用收货地址 ID
- 一个可下单商品 ID
- 对应商家 ID
- 商品金额

## 1. 支付幂等与库存烟测

脚本： [scripts/smoke-verify.ps1](../scripts/smoke-verify.ps1)

用途：

1. 登录拿 token
2. 创建订单
3. 查询最新订单
4. 连续支付两次，确认第二次被拦截

示例：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\smoke-verify.ps1 -UserName demo -Password 123456 -PayKey 123456 -ProductId 1 -BossId 2 -AddressId 1 -Money 200
```

重点观察：

1. 第一次支付成功
2. 第二次支付被拒绝
3. 没有因为重复支付再次扣库存或重复改订单状态

## 2. 秒杀并发压测

脚本： [scripts/flash-sale-load.ps1](../scripts/flash-sale-load.ps1)

用途：并发打秒杀接口，观察防重复和预减库存逻辑。

你可以先通过登录接口拿到 token，再执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\flash-sale-load.ps1 -AccessToken <token> -RefreshToken <token> -ProductId 1 -AddressId 1 -PayKey 123456 -Requests 20
```

重点观察：

1. 同一个用户并发请求时，大部分请求应该被“请勿重复秒杀”挡住
2. 如果换多个用户压同一个商品，成功数不应超过库存数
3. Kafka consumer 应该最终把成功秒杀落成订单

## 3. 手工核对点

建议在压测后手工再查一次：

1. Redis 库存 key 是否与详情缓存库存一致
2. `skill_product2_mqs` 表里是否有异步记录
3. `order` 表里是否生成了对应订单
4. Kafka broker 日志里是否有消费异常

## 4. 当前局限

1. 这两份脚本是手工验证和压测辅助，不是 CI 自动化测试
2. 秒杀压测脚本默认更适合验证“同一用户重复请求”场景；如果要测“多用户抢同一商品”，需要准备多组 token
3. Kafka/Redis/MySQL 没起时，脚本不会替你兜底初始化环境
