# 🛒 gin-mall 项目深度理解指南

> 适合人群：刚学完 Gin / GORM / gRPC / go-zero 的 Go 初学者，想通过此项目找工作 / 复刻实现。

---

## 📐 一、整体架构（先建立全局观）

这是一个 **典型的分层架构** Go 电商后端：

```
请求 → middleware → routes → api(controller) → service → repository(dao/cache/es/mq) → DB
```

| 层        | 职责                    | 对应目录                         |
| --------- | ----------------------- | -------------------------------- |
| 入口层    | 启动、加载配置          | `cmd/`, `config/`                |
| 路由层    | URL 映射                | `routes/`                        |
| 中间件    | 鉴权、跨域、链路追踪    | `middleware/`                    |
| 接口层    | 参数绑定、调用 service  | `api/v1/`                        |
| 业务层    | 核心业务逻辑            | `service/`                       |
| 数据层    | 数据库、缓存、ES、MQ    | `repository/`                    |
| 模型层    | 表结构、DTO             | `repository/db/model/`, `types/` |
| 工具层    | JWT、加密、邮件、上传等 | `pkg/utils/`                     |
| 常量/错误 | 错误码、业务常量        | `consts/`, `pkg/e/`              |

### 阶段 2：用户模块（最简单，从这里入手）⭐

**入口文件链路**：

- [api/v1/user.go](../api/v1/user.go)
- [service/user.go](../service/user.go)
- [repository/db/dao/user.go](../repository/db/dao/user.go)

**功能点**：注册、登录、更新资料、关注/取关、上传头像、邮箱验证

**知识点**：

- Gin：`ShouldBind` 参数校验、`validator/v10` tag
- GORM：`Create / First / Update / Where`、模型钩子（`BeforeCreate` 加密密码）
- **bcrypt** 密码加密（`golang.org/x/crypto`）
- **JWT** 签发与校验：[pkg/utils/jwt/jwt.go](../pkg/utils/jwt/jwt.go) + [middleware/jwt.go](../middleware/jwt.go)
- **Redis 缓存**：[repository/cache/](../repository/cache/)
- **邮件发送**：[pkg/utils/email/email.go](../pkg/utils/email/email.go)（gomail）
- **文件上传**：本地 + 七牛云 OSS [pkg/utils/upload/](../pkg/utils/upload/)
- **Context 传递用户信息**：`ctl.GetUserInfo(ctx)`

> 💡 把这一块吃透，整个项目的 80% 套路你就掌握了。

### 阶段 5：订单 & 支付模块（核心业务）⭐⭐

**文件**：

- [api/v1/order.go](../api/v1/order.go), [service/order.go](../service/order.go)
- [api/v1/pay.go](../api/v1/pay.go), [service/payment.go](../service/payment.go)
- [api/v1/money.go](../api/v1/money.go)

**当前已实现**：下单、订单列表、订单详情、删除订单、支付、金额展示、发货、收货、支付幂等、库存并发控制、支付成功事件发布

**当前代码里能直接学到的点**：

- **GORM 事务**：支付时把买家扣款、卖家加款、库存扣减、订单状态更新放在一个事务里
- **支付幂等**：通过订单状态条件更新，避免重复支付时重复扣款
- **库存并发控制**：通过 DB 条件更新直接扣库存，而不是先查再改
- 订单状态流转：待支付 → 已支付 → 已发货 → 已收货
- **Redis 有序集合**：订单创建后把订单号写入 Redis，记录 15 分钟超时点
- **RabbitMQ 事件**：支付事务完成后异步发布订单支付成功事件
- 订单号生成（时间种子 + 9 位随机数 + 商品 ID + 用户 ID）
- 金额加解密与余额更新流程

**当前还可以继续扩展的点**：

- RabbitMQ 消费端业务，目前仓库里只补了生产端
- 订单超时关闭任务，目前 Redis 只记录超时点，没有后台关闭流程
- 更细的支付失败补偿与审计日志

**学习建议**：

先把 [service/payment.go](../service/payment.go) 和 [service/order.go](../service/order.go) 走通，再看 [repository/db/dao/order.go](../repository/db/dao/order.go) 和 [repository/db/dao/product.go](../repository/db/dao/product.go) 里的条件更新。这里重点学的是事务边界、状态机、幂等和库存一致性，然后再看为什么 RabbitMQ 事件要放在事务提交后。

### 阶段 6：秒杀模块（高并发亮点）⭐⭐⭐

**文件**：

- [api/v1/flash_sale.go](../api/v1/flash_sale.go)
- [service/flash_sale.go](../service/flash_sale.go)
- [repository/db/dao/flash_sale.go](../repository/db/dao/flash_sale.go)
- [repository/db/model/flash_sale.go](../repository/db/model/flash_sale.go)

**当前已实现**：秒杀商品初始化、列表读取、详情读取、缓存预热、Redis 预减库存、防重复秒杀、Kafka 异步下单

**当前代码里能直接学到的点**：

- Redis 列表缓存和详情缓存的预热、回源
- **Lua 脚本**：把重复购买校验和库存扣减收敛成一个原子操作
- 用户购买标记 key、库存 key、详情 key 的配合方式
- **Kafka 异步下单**：先在 Redis 抢资格，再投递 MQ，最后由消费者落库建单
- 失败回滚：Kafka 发布失败时回滚 Redis 预减结果

**当前还可以继续扩展的点**：

- 秒杀结果查询接口，让前端轮询异步下单结果
- 多用户压测脚本和更标准的限流/熔断
- Kafka 消费失败重试、死信队列、监控告警

**学习建议**：

不要只盯着 Kafka。先把 [service/flash_sale.go](../service/flash_sale.go) 里缓存预热、回源、Lua 预减、失败回滚这四段读顺，再去看 [repository/kafka/common.go](../repository/kafka/common.go) 里 producer 和 consumer 怎么把“抢到资格”和“最终建单”拆开。

## 🎯 四、具体行动建议

1. **先跑起来**：用 `docker-compose up` 把依赖起起来，能注册登录是第一个里程碑。
2. **抓一条线吃透**：从「用户注册」开始，跟着调用栈一路点进去，每一行都搞懂。
3. **画时序图**：对秒杀、下单这类复杂流程，画出 `用户 → API → Service → Redis → Kafka → Consumer → DB` 的时序图。
4. **边读边记笔记**：每个模块写一篇"它做了什么 + 用了什么技术 + 为什么这么设计"。
5. **重写时改进**：
   - `dgrijalva/jwt-go` 已废弃 → 升级 `golang-jwt/jwt/v5`
   - Go 1.18 较老 → 升级到 Go 1.21+
   - 引入 `wire` 做依赖注入
   - 用 `slog` 替代 logrus
6. **前端推荐技术栈**：
   - PC 端：Vue3 + Vite + TypeScript + Pinia + Element Plus
   - 移动端：Vue3 + Vant，或 Uniapp
   - 备选：Next.js + Tailwind CSS
